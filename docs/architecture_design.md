# AgentFlow 平台企业级架构设计文档

## 1. 项目背景与目标

### 1.1 业务痛点
* **工具分散**：企业内部存在 100+ 独立脚本和 AI 工具，用户需要手动调用多个工具完成任务。
* **任务复杂**：如「查询某客户最近3个月的订单并生成月度报表」，需要调用订单系统、BI 工具、邮件工具等 5+ 个工具。
* **缺乏编排**：任务依赖关系、失败重试、错误处理均需人工干预。
* **知识孤立**：企业内部知识库分散在 Wiki、文档、数据库中，无法快速检索。

### 1.2 建设目标
重新设计 AgentFlow，目标是打造一个基于 LLM 驱动的**任务拆解、工具调用、记忆管理**的 AI Agent 编排平台。
* **服务对象**：业务部门（处理重复任务）、技术团队（快速开发 AI 工具）、管理员（监控与权限管理）。

---

## 2. 系统架构设计

### 2.1 核心服务模块拆分
平台采用分布式/微服务化架构思想设计（当前 MVP 阶段采用单体实现）：
1. **AgentFlow-Gateway**：统一接入、鉴权、限流。
2. **AgentFlow-Orchestrator**：核心编排引擎，负责 Agent 编排、任务拆解、工具调用。
3. **AgentFlow-Memory**：记忆服务，管理短期记忆（Redis）与长期记忆（向量数据库）。
4. **AgentFlow-RAG**：知识库管理、向量检索、RAG 服务。
5. **AgentFlow-Tools**：工具适配器，对接外部数据库、API、文件系统等。
6. **AgentFlow-Monitor/Admin**：Agent 监控、审计日志、告警与管理控制台。

## 3. 核心业务流程全链路

### 3.1 流程一：Agent 任务执行全链路
1. **网关接入**：用户输入自然语言任务，网关进行鉴权与限流（令牌桶）。
2. **加载配置**：Orchestrator 接收任务，加载 Agent 配置（工具列表、系统 Prompt）。
3. **任务拆解**：调用 LLM，解析返回的工具调用 JSON（决定需要调用哪些工具）。
4. **工具调用（循环执行）**：系统根据解析结果，依序调用后端业务系统 API（如 CRM、订单系统、BI 工具）。
5. **结果汇总**：将所有工具返回的数据拼装，再次调用 LLM 生成最终人类友好的回答。
6. **记忆存储**：将对话存入短期记忆，关键信息存入长期记忆。
7. **触达用户**：返回结果给前端或异步发送通知。

### 3.2 流程二：ReAct 循环 (Reasoning and Acting)
Agent 必须严格遵循 `Thought -> Action -> Observation` 的推理模型：
* **第 1 轮** -> `Thought`: 需要查询客户信息；`Action`: 调用 getCustomer 工具；`Observation`: 返回客户信息。
* **第 2 轮** -> `Thought`: 需要查询订单；`Action`: 调用 getOrders 工具；`Observation`: 返回订单列表。
* **第 3 轮** -> `Thought`: 需要生成报表；`Action`: 调用 generateReport 工具；`Observation`: 返回报表 URL。
* **第 4 轮** -> `Thought`: 已获得所有信息，生成最终回答；`Action`: 无 (Finish)。

### 3.3 流程三：RAG 检索流程
1. **问题 Embedding**：将用户提问调用 Embedding 模型生成 768 维向量。
2. **向量检索**：查询 PostgreSQL + pgvector，使用余弦相似度计算获取 Top-5 相关文档片段。
3. **构建 Prompt**：组装 `System Prompt` + `检索到的上下文` + `用户提问`。
4. **LLM 生成**：基于上下文回答问题。

## 4. 核心底层逻辑实现标准

### 4.1 记忆管理系统 (短期 + 长期)
* **短期记忆 (Redis)**：防止 LLM 上下文窗口耗尽。
  * **滑动窗口机制**：每次存入新消息时，必须执行类似 `trim(-20, -1)` 的操作，**严格保留最近 20 轮对话**。
  * **自动过期**：Session 记忆默认 24 小时过期。
* **长期记忆 (pgvector)**：
  * 使用 PostgreSQL 的 `pgvector` 插件进行存储。
  * 检索核心 SQL 范式：`SELECT content, 1 - (embedding <=> ?) as similarity FROM knowledge_base ORDER BY embedding <=> ? LIMIT TopK`。

### 4.2 工具校验与幻觉防御 (Tool Validation)
为解决 LLM 幻觉导致工具参数错误（成功率低）的问题：
1. **Schema 强化**：定义工具时必须提供详尽的参数描述和类型说明。
2. **参数预校验**：在真正执行业务工具前，必须经过 Validator 拦截器（校验必填项、数据类型匹配等）。
3. **自修复重试**：校验失败抛出异常后，捕获该异常并将错误信息（如 "Missing required parameter"）作为反馈喂给 LLM，触发 LLM 的自我纠错机制（最大重试 3 次）。

### 4.3 死循环熔断机制 (Circuit Breaker)
为防止 Agent 因工具调用失败陷入无限循环重试，导致 CPU 100% 和 API 费用爆炸，必须在 ReAct 主循环中加入**双重熔断防线**：
1. **最大轮次限制**：记录循环次数（如 `int round = 0`），一旦 `round >= 50`（MVP 阶段建议设为 5），立刻抛出 `MaxRoundsExceededException`。
2. **绝对超时截断**：记录任务开始时间，每次循环检查耗时，若超出 10 分钟（MVP 阶段建议设为 3 分钟），立刻抛出 `TimeoutException`。

### 4.4 敏感数据脱敏与审计 (DLP)
为防止 Agent 在对话中泄露用户隐私信息：
1. **正则脱敏**：在 Agent 输出最终结果前，使用正则表达式匹配并替换敏感数据（例如将身份证、手机号替换为 `***`）。
2. **强制审计**：调用 `AuditLogService`，将**用户原始输入**、**Agent 原始输出 (Raw Output)** 以及**脱敏后的输出 (Filtered Output)** 一并落库记录，满足企业级安全合规要求。

### 4.5 并发调度与死锁防范
当多 Agent 并发调用同一核心资源时：
1. **分布式锁**：调用资源前尝试获取锁（如 Redis 分布式锁 `tryLock`）。
2. **超时强制释放**：锁必须带有超时时间（如 5 分钟），防止 Agent 宕机导致死锁。
3. **安全释放**：释放锁时必须确保 `isHeldByCurrentThread()`，防止误释放其他 Agent 的锁。
